Module 01
MSBTE Central Backend — Authentication, COB, DEC & Integration Services
The primary application server powering login and secure session management for all user roles — Administrators, Institutes, RBTE Officers, and Candidates — alongside Change of Branch (COB), Diploma Engineering Course (DEC), Circulars, Digital Evaluation, and MahaIT/RTS integration services
CCommercial / Functional Changes — Authentication & Identity
  • Unified login engine for all roles: A single, centralised authentication service now handles login for all user types — Administrators, Institutes, RBTE Officers, and Candidates. Each role is granted a secure session token that controls precisely what that user may access across the entire platform.
  • Password management self-service: Users can change or reset their own passwords and update their registered mobile number through dedicated, secure flows — reducing support requests for credential management.
  • OTP-based mobile verification: Mobile number updates require a one-time password (OTP) verification step, ensuring only the verified device owner can update contact details.
  • Admin-controlled default password generation: Administrators can generate default passwords for new users through a separately secured administrative route — preventing abuse by non-admin roles.
  • Login rate limiting: Repeated failed login attempts from the same source are automatically throttled — the system responds with a clear message indicating when the user may try again, protecting accounts against automated brute-force attempts.
CCommercial / Functional Changes — COB, DEC & Operations
  • Change of Branch (COB) at admission and institute levels: Candidates can apply for a change of branch at admission stage, selecting their preferred discipline with a structured application flow (apply, review details, confirm, cancel). Institutes can also initiate and manage COB applications on behalf of candidates within their jurisdiction.
  • Diploma Engineering Course (DEC) management: RBTE officers can manage DEC registrations for 1st, 2nd, and final-year students — adding, viewing, and removing registrations with validated input and RBTE-gated access ensuring only authorised officers can perform these operations.
  • Circulars and operational notifications: The system manages access to circulars for different user roles, including restricted circulars accessible only to authorised parties. Malpractice-related circulars and authority letters are handled through dedicated, access-controlled routes.
  • Question Paper (QP) Statistics: Administrators can access statistical reports on question paper usage and distribution — supporting examination paper management without requiring direct database access.
  • Digital evaluation activity tracking: Institute-level digital evaluation progress can be monitored through dedicated activity endpoints, giving administrators and RBTE officers a real-time view of evaluation completion status.
  • ABC ID (Aadhaar-Based Credit) operations: The backend validates and checks the existence of a candidate's Aadhaar-linked ABC ID as required for academic credit transfer compliance.
  • SC/DC distribution and dispatch reporting: Comprehensive reporting on the distribution of answer books from Sub-Centers (SC) to Distribution Centers (DC) and onward — with multiple dispatch variants (SC lead, controller to DC) and correct ordering and empty-result handling.
  • Online exam center management: Pre-computed data paths for online exam center details reduce real-time database load during high-concurrency periods such as hall ticket generation and seating allocation.
  • Affiliation status handling: The system correctly communicates an institute's affiliation status — including returning a clear, standardised response when an affiliation is blocked — preventing ambiguity in downstream institute-facing operations.
CCommercial / Functional Changes — MahaIT & RTS Integrations
  • MahaIT integration updated: The service URLs connecting to the Maharashtra government's MahaIT platform have been updated to reflect current endpoint addresses — ensuring the application tracking and related government-linked services continue to work without interruption.
  • Application tracking messaging improved: The status messages displayed to candidates tracking their application status through the MahaIT link have been made clearer and more informative, reducing confusion about where a candidate's application stands in the process.
  • RTS dashboard integration: The MSBTE platform connects to the Real-Time Service (RTS) dashboard used by government monitoring authorities. The integration has been updated to correctly pass required parameters and handle date-based filtering for accurate reporting.
  • Environment-driven database routing: Which database is used for a given operation (main, e-marksheet, or non-AICTE) is now determined entirely by environment configuration — this makes it straightforward to point different environments (development, staging, production) at the appropriate databases without code changes.
BBusiness / Operational Impact
  • All roles login through one secure gateway: A single authentication engine reduces inconsistency in how different user types are verified — every login, regardless of role, goes through the same secure session management process with role-appropriate access.
  • Account brute-force protection is now active: Automated login-attempt throttling is in place for all roles — administrators no longer need to manually intervene when suspicious login patterns are detected.
  • COB applications are fully digital and traceable: The Change of Branch process — from application to confirmation — is now entirely managed through the platform, with a clear audit trail at every step.
  • RBTE DEC operations are governed by access control: Only authorised RBTE officers can manage DEC registrations, and all operations are validated — reducing the risk of incorrect registrations entering the official records.
  • Government platform connectivity maintained: MahaIT and RTS integrations are kept current — MSBTE's compliance with state government reporting requirements is not disrupted by infrastructure changes at the government end.
  • Stationery dispatch reporting is accurate: SC/DC answer book distribution reports now reflect correct ordering and handle scenarios where no data exists (returning an empty list rather than an error), giving operations teams a reliable view of dispatch status.
TTechnology Platform
  • Application server: Node.js with Express 5 — a modern, high-performance web framework serving concurrent requests from all role types simultaneously.
  • Database: MySQL as the primary data store, with query maps centralising all database operations. Advanced SQL techniques (Common Table Expressions for SC center queries, parallel retrieval for online exam center data) are used where performance is critical.
  • Session security: All sessions are managed using industry-standard JWT (JSON Web Token) technology — tokens are cryptographically signed and include role and expiry information, so the server can verify identity and access rights without storing session state.
  • Input validation: All API inputs are validated using Joi schemas — structured rules covering data types, formats, and allowed values — before any business logic is executed.
  • Security layer: HTTP security best practices are enforced at the server level (secure headers via Helmet, CORS origin allowlist, rate limiting at both global and login-specific levels) — these apply to every request regardless of role.
  • Mixed payload handling: The server correctly handles both standard JSON requests and specialised encrypted payloads used by certain MahaIT/RTS integration paths — without one type interfering with the other.
ASystem Design Improvements
  • Role-based route trees (Admin, Institute, RBTE, Candidate, Auth, Common, MahaIT) keep each role's operations cleanly separated — changes to one role's routes do not affect others.
  • All database operations follow a consistent controller → service → query map pattern — improving maintainability and making SQL improvements easier to apply uniformly.
  • Centralised configuration management (CircularManager, QueryMap centralisation) ensures that changes to business rules are made in one place and consistently applied across all relevant operations.
  • Rate limiting is configured at both global and login-specific levels — general API protection and account security protection are independently tunable.
  • Maintenance window middleware allows the platform to be placed in a controlled maintenance state without a deployment, displaying a clear message to all users.
SSecurity Enhancements
  • JWT-based session management: All sessions carry cryptographically signed tokens with role and expiry — every request is verified against the token before processing.
  • Login rate limiting: Repeated failed login attempts are throttled with a clear retry-after response — protecting all user accounts against automated attack.
  • Origin allowlist (CORS): Only requests from approved domain origins are accepted — browsers and automated clients from unapproved origins receive an explicit rejection.
  • Secure HTTP headers: Helmet enforces a deny-frame policy and other security headers on every response — mitigating common web security vulnerabilities.
  • Data encryption in database: Mobile number fields in the database use AES encryption — sensitive contact information is not stored in plain text.
  • Controlled file access: File downloads (such as DEC manuals) use sanitised filename handling — preventing unauthorised file path traversal.
XChallenges & Solutions
Challenge
The MahaIT integration sends data in an encrypted format that cannot be parsed as standard JSON — the general request processing layer was rejecting these payloads, causing integration failures.
Solution
The server now intelligently identifies MahaIT-specific request paths and processes them with a separate plain-text parser, leaving all other endpoints unaffected. Encrypted payloads are handled correctly without disrupting the rest of the platform.
Challenge
Database table names for certain modules (RAC malpractice, copy case, online exam) were inconsistent across environments — code referencing the wrong table name caused silent data errors or query failures.
Solution
Table names have been standardised and aligned across all environments. Where names are environment-specific (e.g., different tables per examination season), they are now driven by configuration variables rather than hardcoded values.
Challenge
SC/DC answer book distribution queries were returning duplicate rows and incorrect ordering — making the dispatch reports unreliable for operations staff verifying stationery distribution.
Solution
Distribution queries were rewritten with correct grouping, ordering, and de-duplication logic. Scenarios where no dispatch data exists now return a clean empty list rather than an error — so the reports are reliably accurate in all cases including early in the examination cycle.
Challenge
The platform operates behind a reverse proxy (a common enterprise network configuration), which was causing IP addresses used for rate limiting and audit logs to reflect the proxy rather than the actual user.
Solution
The server is now configured to trust the forwarded IP address from the proxy — ensuring rate limiting and access logs always reflect the real client's identity.

Module 02
MSBTE Admin Dashboard
Central administration and operator control panel for managing all examination-related activities across RBTE, institutes, and internal teams
CCommercial / Functional Changes
  • Summer 2026 exam cycle readiness: All internal configurations updated to align with the Summer 2026 examination cycle — exam numbers, timetable references, marksheet tables, and seating data sources migrated to the new season.
  • Seating chart management: Full seating arrangement workflows introduced — institute-level and RBTE-level seat allocation, NIL (no candidate) handling, corrections, and official confirmations.
  • ABC ID management: Added reporting and statistical dashboards for Academic Bank of Credits (ABC) ID tracking; photo correction and bulk/individual photo download tools introduced.
  • E-Marksheet & distribution reports: New RBTE-facing reports for e-marksheet status, Distribution Center (DC) – Exam Center (EC) – Chief Inspector (CI) module, and sheet distribution.
  • Affiliation administration expanded: New workflows for onboarding institutes, managing affiliation removal, viewing affiliation history, course details, and add-course requests.
  • Enrollment management: Candidate search by enrollment number and date-wise confirmation tracking added for operators.
  • Circular management: Ability to publish, toggle active/inactive, and manage circulars for EC, DC, and SC levels from a single interface.
  • RBTE operational reports: Comprehensive new reports covering fee collection, COB (Cut-Off Batch), inventory, detained candidates, malpractice cases, and remuneration revisions.
  • Bulk communications: Bulk email dispatch with editable templates updated for Summer 2026 examination cycle communications.
  • Bulk candidate registration: Improved batch registration with audit logging and safeguards against duplicate processing.
  • Hall ticket & DTE data preparation: Workflows for preparing hall ticket data and submitting information to DTE (Directorate of Technical Education).
  • Two-factor authentication (2FA): All administrator accounts now require a one-time passcode from an authenticator app in addition to the regular password — significantly raising account security.
  • Module Control Panel: Administrators can now toggle the visibility of individual module links (e-marksheet, exam form, COE, etc.) for specific roles without requiring any technical deployment.
BBusiness / Operational Impact
  • Extended session timeout: Operator auto-logout threshold increased from 15 minutes to 50 minutes, reducing interruptions during complex report generation or data entry tasks.
  • Season transition made error-free: All season-specific data references (Winter 2025 → Summer 2026) are now managed from a single central configuration file, eliminating manual errors that occurred when updating dozens of scattered references.
  • Report usability improvements: Print-friendly layouts, better export options, and improved navigation across all reports reduce the time operators spend on repetitive tasks.
  • Zero-code operational control: Features such as opening/closing module links can be toggled by non-technical operators without involving the development team.
  • Stronger administrator accountability: Role-restricted access ensures that sensitive reports are visible only to the appropriate designation, improving audit readiness.
TTechnology Platform
  • Application framework: Python-based web application (Django) with an established, industry-standard architecture maintained on latest stable versions.
  • Background task processing: Heavy or time-consuming operations (report generation, bulk registration, data preparation) run as background tasks, keeping the main portal responsive.
  • Database: MySQL — a mature, production-grade relational database. All queries optimised for the Summer 2026 schema.
  • Security tokens: All sensitive report endpoints are protected by cryptographically signed access tokens that embed the user's role — preventing unauthorised access even if a link is shared.
  • AI-assisted search (Pilot): A conversational AI module has been introduced as a pilot feature, allowing administrators to query operational data using plain-language questions.
  • Automated deployment pipeline: Code changes are automatically tested and deployed to the server, reducing manual deployment steps and human error.
ASystem Design Improvements
  • Functionalities organised into distinct, independently maintainable units (exam, affiliation, seating, enrollment) — easier to extend without breaking existing features.
  • Background jobs handle large data operations asynchronously, ensuring the UI remains fast for other users during peak processing.
  • Feature switches allow enabling or disabling portal sections without modifying application code.
  • All season-specific configuration centralised — adding a new exam season requires changing one file, not dozens.
SSecurity Enhancements
  • Two-factor authentication (2FA): Admin accounts now require a second verification step via authenticator app — protects against compromised passwords.
  • Role-based report access: Reports are restricted to specific user roles; no user can access data beyond their assigned designation.
  • Session security: Response headers hardened on password-related pages to prevent data leakage; API responses carry appropriate cache-control directives.
  • Input validation fix: A data query issue in the ABC ID lookup was corrected, preventing potential over-fetching of records.
XChallenges & Solutions
Challenge
Every new exam season (e.g. Summer 2026) required updating dozens of table and database references scattered across the entire application, making it error-prone and time-consuming.
Solution
All season-specific names are now stored in a single central configuration file. Transitioning to a new season requires updating only that one file, and all reports and workflows automatically use the new values.
Challenge
Bulk affiliation enrollment operations sometimes failed silently due to strict database validation rules conflicting with batch data formats.
Solution
Batch write operations are now handled with appropriate safeguards — data is validated, temporarily relaxed constraints are applied only where safe and necessary, and all inserts are logged for auditability.
Challenge
A change-of-branch (COB) operation in scheme updates could cause certain enrollment records to get stuck in a processing loop, consuming server resources.
Solution
Background task logic updated to detect and skip already-processed or inapplicable records, preventing infinite loops and freeing up server resources.

Module 03
Exam Form & Sub-Modules — Application Processing Engine
Server-side engine processing exam form applications for Candidates, Institutes, and RBTE — extended to cover Copy Case, Hall Ticket, Scholarship, Stationery, Seating Chart, and Attendance sub-modules
CCommercial / Functional Changes — Core Exam Form
  • Complete elective subject recording: Previously, only subjects a candidate actively selected were saved. Now, both chosen elective subjects and the remaining optional subjects are recorded together — ensuring no elective data is ever missing from examination records or downstream result processing.
  • Safe re-submission without duplicates: When a candidate re-submits or an operator reprocesses a form, the system safely archives the previous version and starts fresh — eliminating the need for manual database cleanup after re-runs.
  • RBTE examform governance: RBTE officers can only fill exam forms for institutes that are on an explicit approved whitelist, and only during the configured operational window. Requests outside the window or from unlisted institutes are rejected immediately with a clear access-denied response.
  • Operational window control: Exam form submission, confirmation, and cancellation are controlled by a central configuration switch — opening or closing the form for any user group requires only a configuration change, with no deployment needed.
  • Distinct workflows by user role: Candidates, institutes, and RBTE staff each follow a separate, tailored workflow with appropriate data access at every step.
CSub-Module Changes — Copy Case (Malpractice)
  • Malpractice records corrected: A data join error that caused malpractice subject records to match the wrong paper or candidate has been resolved — every malpractice record is now correctly linked to the exact seat, paper, and serial number of the candidate involved.
  • Exam center jurisdiction fix: Malpractice lookups now correctly filter by the connected institute or distribution center as the operational jurisdiction — preventing cases from the wrong center appearing in a center's malpractice list.
  • RBTE-side malpractice operations: RBTE can now view, distribute, and check the status of copy cases across centers from a dedicated set of operational endpoints, separate from institute-level workflows.
CSub-Module Changes — Hall Ticket
  • Multiple authentication modes supported: Hall ticket access is handled differently depending on the context — kiosk/public print mode uses a separate verification pathway, while logged-in institute and RBTE users use their standard session credentials. This allows hall tickets to be securely accessible in both self-service and supervised contexts.
  • Online exam center code consistency: The exam center code displayed on hall tickets has been aligned with the official online exam center records, resolving discrepancies that were causing hall tickets to show an incorrect or outdated center.
  • Institute and RBTE bulk access: Institutes and RBTE officers can retrieve multiple hall tickets in a single request, significantly reducing the time needed to print hall tickets for an entire batch of candidates.
CSub-Module Changes — Scholarship
  • Institute scholarship workflows: Institutes can fetch candidate scholarship details, submit applications, look up bank master data, and manage required documents — all through a consistently secured set of endpoints shared with RBTE verification flows.
  • Autonomous vs non-autonomous separation: RBTE scholarship operations are split into two clearly distinct workflows — one for autonomous institutes and one for non-autonomous institutes — preventing data from one stream appearing in the other's verification queue.
CSub-Module Changes — Stationery (Winter & Summer)
  • Serial number tracking introduced: Distribution Center and Exam Center stationery allocation now includes structured serial number range management — operators record the exact range of serial numbers dispatched to each center, with a confirmation step to finalise the allocation.
  • Separate winter and summer stacks: Winter and Summer examination stationery workflows are handled by distinct route configurations — operational changes for one season cannot accidentally affect the other.
  • Reduced manual reconciliation: The structured serial number confirmation workflow provides an auditable record of every stationery dispatch, significantly reducing the manual effort previously required to reconcile DC/EC stationery counts at the end of each examination session.
CSub-Module Changes — Seating Chart & Attendance
  • Seating chart correction validation: Input validation for seating chart corrections has been tightened — incorrect or incomplete correction requests are now rejected before being saved, preventing bad seating data from propagating to examination day operations.
  • NIL reporting for seating: RBTE can now generate NIL (no candidate) reports for seating positions that were allocated but left vacant — providing a complete picture of actual seat utilisation per center.
  • Attendance course and block generation: Institutes can retrieve the list of courses eligible for attendance marking and generate attendance blocks for each course — supporting structured attendance management during examination days rather than manual list creation.
BBusiness / Operational Impact
  • No missing elective data in examination records: Complete elective data (both chosen and optional subjects) is now guaranteed in every exam form — resolving a persistent gap that affected downstream reporting and result generation.
  • Zero manual cleanup on reprocessing: Batch re-runs are fully safe — the system archives the previous version and starts clean, requiring no manual database intervention.
  • RBTE examform access secured by whitelist: Unauthorised RBTE fill operations are blocked at the server level with no data reaching the database — reducing the risk of accidental or unauthorised examination form manipulation.
  • Stationery dispatch is now fully auditable: Every stationery allocation is recorded with serial number ranges and confirmed by the dispatching center — providing an end-to-end paper trail for examination stationery management.
  • Malpractice records are reliably accurate: Join key corrections mean the right candidate's paper is always linked to the right malpractice case — a critical accuracy requirement for examination integrity.
TTechnology Platform
  • Application server: Node.js with Express 5 — current-generation, high-performance web framework handling concurrent requests from candidates, institutes, and RBTE simultaneously.
  • Database: MySQL — all subject records, form status, stationery allocations, malpractice cases, and audit copies stored with structured query patterns. Backup tables preserve a full history of re-applied forms.
  • Access security: Every endpoint validates the caller's role (candidate, institute, RBTE, or joint institute-RBTE access) via cryptographically signed tokens before processing any request.
  • Input validation: Strict validation rules applied to all submissions — enrollment format, subject codes, serial number ranges, seat identifiers, and exam window codes — before any data reaches the database.
  • In-memory lookup caching: Frequently referenced data (exam center codes, paper block mappings) is cached in memory during a request, reducing repeated database reads during peak concurrent submissions.
ASystem Design Improvements
  • Each role (candidate, institute, RBTE) has its own independent route tree — changes to one role's workflow cannot break another's.
  • All database queries centralised in shared query libraries — fixes and improvements apply consistently across all form types and roles.
  • Sub-modules (copycase, hallticket, scholarship, stationery, seating, attendance) each have their own dedicated controller-service-query structure, keeping concerns cleanly separated.
  • Shared utilities handle common operations (exam center resolution, paper block lookup) across all sub-modules without code duplication.
SSecurity Enhancements
  • Role-based access enforcement: Candidate, institute, and RBTE operations strictly separated — no user can access data or perform operations outside their designated role.
  • RBTE whitelist enforcement: RBTE exam form fill operations are further restricted to an explicit approved list of RBTE codes — operations from any unlisted code are rejected outright.
  • Hard feature window enforcement: All exam form, scholarship, and stationery operations are blocked outside configured operational windows — no data is written to the database on out-of-window requests.
  • Strict input validation: Joi validation schemas applied to all submission payloads — incorrect formats are rejected before reaching database logic.
XChallenges & Solutions
Challenge
Malpractice subject records were being matched using only the paper code, causing cases from different candidates sharing the same paper to be incorrectly linked — a serious examination integrity risk.
Solution
The serial number is now included as a join key in all malpractice-related queries — ensuring every malpractice record is uniquely matched to the exact candidate, seat, and paper combination.
Challenge
Exam center jurisdiction for malpractice and hall ticket lookups was using an imprecise filter, causing records from outside a center's actual jurisdiction to appear in that center's lists.
Solution
Filter criteria updated to use the precise operational jurisdiction identifiers (connected institute and distribution center codes), ensuring each center sees only its own candidates and cases.
Challenge
RBTE officers could trigger exam form fill operations for any institute without restriction, creating a risk of unauthorised form processing outside officially sanctioned operations.
Solution
An explicit RBTE-code whitelist is now enforced on all RBTE exam form fill routes — only pre-approved RBTE codes can initiate fill operations, and only within the configured operational window.

Module 04
Theory E-Marksheet — Institute Marks Entry Portal
Web portal used by institutes for digital theory marks entry, automated marksheet generation, answer-book receipt management, and digital evaluation reporting
CCommercial / Functional Changes
  • Institute-first redesign: The portal has been restructured around the institute's role as the primary operator — navigation, workflow steps, and page labels now reflect institute responsibilities rather than individual examiner tasks.
  • Digital evaluation suite: Institutions can now manage automated e-marksheets, create manual examiner marksheets, track paper codes, and record receipt of answer books — all from a single, unified interface.
  • Sheet status reports: New reports identify answer sheets that have been physically filled by examiners but not yet digitally confirmed, helping RBTE and institutes follow up on pending confirmations promptly.
  • Special marks code enforcement: The list of permissible special marks codes (used to indicate situations such as absent, malpractice, or medical grounds) has been refined and locked to prevent invalid entries, with clear on-screen guidance for operators.
  • Summer 2026 academic year updates: All season labels, report headers, and application paths updated to reflect the Summer 2026 examination cycle.
  • Configurable dashboard display: The number of records shown per page, the visibility of specific features (paper code lists, answer-book receipts, digital evaluation status), and cancellation rules can all be adjusted without changing the application code.
  • Improved sign-in experience: Institute login now validates the correct format of the institute code at entry; users are guided directly to the password reset flow if an unrecognised username is entered.
BBusiness / Operational Impact
  • Fewer data entry errors: Marks entry is locked to read-only once confirmed, and special codes are restricted to only the approved set — eliminating a category of incorrect data entry that required manual correction.
  • Faster issue resolution: Progress indicators (loading bars) during data retrieval and clear success/failure notifications on marks submission allow operators to immediately know whether an action succeeded, reducing calls to support.
  • Correct receipt table totals: Fixes to how special-code counts and answer-book totals were calculated in the Receipt Summary report ensure that what operators see matches the actual data.
  • Automated deployments: The portal is automatically tested and deployed to development, staging, and live environments upon each update — eliminating manual deployment steps and reducing the risk of errors during go-live.
TTechnology Platform
  • Modern web application: Built with React 18 — a widely adopted, component-based UI framework — providing a responsive, fast-loading interface on both desktop and tablet browsers.
  • Optimised data loading: A centralised data-fetching layer ensures that data is loaded once and reused across different parts of the application, reducing redundant server requests and improving page load times.
  • PDF generation: Reports and marksheets can be generated and downloaded as PDF documents directly from the browser without requiring a separate print application.
  • Excel export: Key reports can be exported to Excel for offline analysis and record-keeping by institute staff.
  • Component library updated: The UI toolkit (Material UI) and build tools have been kept on current stable versions, ensuring continued security patch coverage.

Module 05
Non-Theory E-Marksheet — HOD, Examiner & Institute Portal
Web portal for non-theory marks entry and confirmation used by Heads of Department, examiners, principals, and institute staff — covering both AICTE and Non-AICTE examination streams
CCommercial / Functional Changes
  • Controlled feature rollout: Individual portal features (HOD login, examiner login, automated e-marksheets, manual e-marksheets, reports, confirmation flows, maintenance window) can now be switched on or off independently for development, staging, and production environments — allowing phased, risk-controlled releases.
